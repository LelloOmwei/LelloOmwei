OMWEI

Execution Integrity Signal

OMWEI is a hardware-backed signal indicating that system execution is enforced and verifiable.

Defined by Equinibrium

Definition

A system is OMWEI-enabled if its execution behavior is constrained and validated at the hardware level.

  • Instruction flow is validated before execution
  • Unsafe or unauthorized behavior is blocked
  • Execution is deterministic and auditable

What It Means

OMWEI replaces software-defined trust with hardware-enforced execution guarantees.

  • No reliance on post-execution validation
  • No dependence on software-only guardrails
  • Execution constraints are enforced in silicon

Implementation

OMWEI is enabled through silicon IP integrated into processor architectures.

  • Instruction-level enforcement logic
  • Constraint-aware execution path
  • Hardware validation of execution context

Example

if (instruction_valid && constraints_ok) { execute(); } else { block_execution(); }

Architecture Specification (Draft v0.1)

OMWEI introduces a pre-execution validation model embedded directly into the processor pipeline.

Fetch → SIGNIFY → Decode → Execute

Problem

Classical processors assume that all input data is valid. This assumption no longer holds in distributed and autonomous systems.

As a result, systems execute invalid or malicious inputs with full determinism.

Model

The SIGNIFY stage introduces deterministic validation before instruction decode.

  • Validation happens before execution
  • Invalid inputs are blocked in hardware
  • No reliance on probabilistic models

Core Components

SLC (Semantic Logic Core)
Hardware validation unit enforcing constraints.

32BSA (Semantic Atom)
Fixed-size structured execution input.

AX Buffer
Memory boundary allowing only validated data.

Execution Model

  • Input is acquired from external sources
  • Validated in SIGNIFY stage
  • Invalid inputs are blocked
  • Only validated data reaches execution

Limitations

  • Requires predefined constraints
  • Does not guarantee correctness of decisions
  • Adds hardware complexity
  • Depends on structured input (e.g. 32BSA)

Integration

OMWEI can be implemented as a co-processor or integrated IP block within existing architectures (RISC-V, ARM).
